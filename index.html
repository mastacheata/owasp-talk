Web Security mit dem OWASP ASVS

Kolloquium zur Masterarbeit "Untersuchung einer Webanwendung nach dem OWASP Application Security Verification Standard" von Benedikt Bauer
OWASP = Open Web Application Security Project OWASP ähnlich GNU, Linux etc., Open Source, loser Zusammenschluss von Menschen Security Firmen und Professionals Entwickeln allerhand Sicherheitskram mit Fokus auf Web
Injection - SQL, Reflection XSS, Code Injection
Broken Auth - Kaputte Passworte, Sesssion Tokens etc.
SDE - DSGVO lässt grüßen
XXE - Hauptsächlich SOAP und SAML, ähnliches evtl. via GraphQL/JSON möglich
Broken Access - Zugriff auf Fremde Daten, Admin für Gäste etc.
Misconfiguration - default configs, open cloud storage, verbose errors including credentials
XSS - Session Hijacking, Cookies auslesen, *coins minen
Insecure Deserialization - JSON mit eval() :P
Known Vulnerabilities - Selbsterklärend
Insufficient Logging - Nach dem Angriff, bleibt Angreifer für durchschnittlich 200 Tage unbemerkt
- Auch Mozilla und Django verweisen auf die Top10 bzw. Testing Guide von OWASP - Wieder nur bekannte Probleme abklopfen, wenig auf allgemeine Sicherheitsmaßnahmen geachtet
- Docker - Single Page App - REST-API - Amazon-Systeme
- SSLLabs von Qualsys zeigt Algorithmen und Konfigurationsprobleme - pyt prüft auf systembefehle und unmaskierte SQL Queries - Linter prüfen CodeQualität, können aber dadurch auch Fehler finden - Linter sind nicht auf Sicherheit fokussiert
- Structure.md, package.json, requirements.txt - Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, Elevation of privilege (STRIDE)
- Empto hat Tokens im Klartext gespeichert - Token IST wie Passwort
- Empto Tokens waren endlos gültig - Für mehrere Sessions zuständig - Nur Abmeldung konnte Token invalidieren
- CSRF - Parameter Manipulation (Alter, Rolle, Firma, etc.) - Trennung von Zuständigkeiten / minimale Berechtigungen
- LFI nur für PHP relevant - XSS usw. durch React bzw. Django ORM - Protokollieren ist schwer - Schema Validierung, bspw. JSON-Schema
- Fehlermeldungen = Padding Oracle Problem - Padding oracle nur für CBC interessant, in TLS 1.2 aber nicht enthalten - Padding oracle nur bei expliziten Fehlermeldungen - openSSL hat zertifizierte Version, muss aber explizit installiert werden - /dev/urandom ausreichend sicher - HSM nicht vorhanden
- Fehlermeldungen sehr generisch - Logs sind auf Sentry - Quelle (Front- / Backend) wegen Vertrauenswürdigkeit der Logs - Nicht-Abstreitbarkeit ist problematisch
- No-Cache Header - Datensparsamkeit eher umgekehrt bei empto - Zugriff wird nicht protokolliert (Access Logs) - BDSG/DSGVO verlangt das aber auch nicht - Vorhaltefrist wird nicht geprüft, Datenlöschung geht nur händisch
- HPKP ist umstritten, weil kompliziert - Besser DANE oder TLS Server Identity Pinnig with Tickets über DNS
- CSP moderner als X-XSS, aber gleiches Ziel - Zeichensatz in Kopfzeile aufnehmen - Content-Sniffing kann Problem für Client sein
- Docker, LoadBalancer zum Abschotten vom Internet
